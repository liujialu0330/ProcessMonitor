## 1. 产品概述

### 1.1 产品定位
进程监控助手是一款专业的Windows桌面应用程序，旨在为软件测试人员提供强大的进程性能监控工具，帮助发现和诊断内存泄漏、CPU占用异常等性能问题。

### 1.2 产品目标
- 提供实时的进程性能指标监控
- 记录并可视化历史监控数据
- 简化测试流程，提高测试效率
- 提供现代化、易用的用户界面

### 1.3 目标用户
- **主要用户**：软件测试工程师
- **次要用户**：性能测试工程师、开发工程师
- **使用场景**：
  - 桌面应用性能测试
  - 内存泄漏检测
  - CPU占用分析
  - 长时间运行稳定性测试

## 2. 核心功能需求

### 2.1 实时监控功能

#### 2.1.1 进程选择
**需求描述**：用户可以通过多种方式选择要监控的进程

**功能要求**：
- 支持通过PID直接输入选择进程
- 支持从系统进程列表中选择进程
- 进程列表支持刷新，获取最新运行进程
- 显示进程名称和PID信息

**优先级**：P0（必须）

#### 2.1.2 指标选择
**需求描述**：提供多种性能指标供用户选择监控，支持单个任务同时勾选多个指标（多指标监控）

**功能要求**：
- 点击"选择指标"按钮弹出多选对话框，指标按分类分组展示
- 每个分类附带三态复选框，支持该分类下"全选/部分选中/取消全选"
- 已选指标数量与名称摘要实时显示在按钮旁（内容过长时省略显示，完整内容通过提示条查看）
- 默认预选"工作集内存"一项，至少需勾选1个指标才能确认
- 一个监控任务可同时采集多个指标，采集时共用同一采集周期与同一时间戳

**支持的指标**（共27个，按分类分组）：

| 分类 | 指标名称 | 说明 | 单位 |
|------|---------|------|------|
| 内存 | 工作集内存 | 进程实际占用的物理内存（RSS/WSET） | KB |
| 内存 | 虚拟内存 | 进程使用的虚拟内存大小（VMS） | KB |
| 内存 | 内存使用率 | 占系统总内存的百分比 | % |
| 内存 | 工作集峰值 | 进程运行期间工作集内存的峰值 | KB |
| 内存 | 专用工作集 | 进程独占、不与其他进程共享的工作集内存 | KB |
| 内存 | 提交大小 | 进程提交的虚拟内存大小（Pagefile） | KB |
| 内存 | 提交大小峰值 | 提交大小的峰值 | KB |
| 内存 | 分页池 | 进程使用的分页池内存 | KB |
| 内存 | 分页池峰值 | 分页池使用量的峰值 | KB |
| 内存 | 非分页池 | 进程使用的非分页池内存 | KB |
| 内存 | 非分页池峰值 | 非分页池使用量的峰值 | KB |
| 内存 | 页面错误 | 累计发生的页面错误次数 | 次 |
| 内存 | 唯一集大小 | 进程独占、不可与其他进程共享的物理内存（USS） | KB |
| CPU | CPU使用率 | 进程CPU占用百分比 | % |
| CPU | CPU用户时间 | 进程在用户态累计消耗的CPU时间 | 秒 |
| CPU | CPU系统时间 | 进程在内核态累计消耗的CPU时间 | 秒 |
| CPU | 优先级 | 进程的调度优先级 | 无 |
| 系统资源 | 线程数 | 进程创建的线程数量 | 个 |
| 系统资源 | 句柄数 | 进程打开的句柄数量（Windows） | 个 |
| 系统资源 | 自愿上下文切换 | 进程主动让出CPU导致的上下文切换次数 | 次 |
| 系统资源 | 非自愿上下文切换 | 因时间片耗尽等被动导致的上下文切换次数 | 次 |
| IO操作 | IO读取字节 | 累计读取的数据量 | KB |
| IO操作 | IO写入字节 | 累计写入的数据量 | KB |
| IO操作 | IO读取次数 | 累计读操作次数 | 次 |
| IO操作 | IO写入次数 | 累计写操作次数 | 次 |
| IO操作 | IO其他字节 | 除读写外其他I/O操作累计字节数 | KB |
| IO操作 | IO其他次数 | 除读写外其他I/O操作累计次数 | 次 |

**优先级**：P0（必须）

#### 2.1.3 采集周期配置
**需求描述**：用户可以自定义数据采集的时间间隔

**功能要求**：
- 采用数值输入框（微调框）直接输入采集周期，不再提供固定预设选项
- 输入范围：1~3600秒的整数
- 默认采集周期为1秒
- 超出范围时自动限制在合法区间内

**优先级**：P1（重要）

#### 2.1.4 多任务监控
**需求描述**：支持同时监控多个进程的不同指标

**功能要求**：
- 最多同时监控5个任务
- 每个任务独立配置监控指标和采集周期
- 达到任务上限时给出明确提示
- 支持单独停止某个监控任务

**约束条件**：
- 最大并发任务数：5个
- 防止系统资源过度占用

**优先级**：P0（必须）

#### 2.1.5 实时数据显示
**需求描述**：以卡片形式实时显示监控数据

**显示内容**：
- 进程名称和PID
- 各监控指标的当前值（带单位格式化，多指标以网格形式逐项排列）
- 已记录的采集次数
- 停止按钮

**更新频率**：根据采集周期实时更新

**优先级**：P0（必须）

### 2.2 历史数据功能

#### 2.2.1 数据持久化
**需求描述**：所有监控数据自动保存到本地数据库

**功能要求**：
- 使用SQLite数据库存储，启用WAL（Write-Ahead Logging）模式提升读写并发稳定性
- 采用批量保存接口写入数据（当前批大小配置为1条，即每次采集立即落盘，以保证历史页面能实时看到最新数据）
- 数据落库失败时自动重试：采集缓冲保留待重试数据（上限1000条，超出丢弃最旧数据），连续失败达到阈值时提示用户一次（v1.2.0）
- 自动管理数据库连接
- 支持长时间监控数据存储

**优先级**：P0（必须）

#### 2.2.2 历史任务查看
**需求描述**：用户可以查看所有监控任务的历史记录

**功能要求**：
- 显示所有包含数据的任务（含运行中和已停止），任务列表在历史数据页与导出数据页统一为"有数据即显示"，不再按进程名等规则额外隐藏（v1.2.0：此前被隐藏的短时/测试任务将重新显示，可通过删除按钮清理）
- 任务信息包含：进程名、PID、指标（单指标显示指标名，多指标显示"N项指标"）、开始时间
- 支持任务选择和切换
- 多指标任务额外提供"指标"下拉框，用于切换查看某一项指标的图表与数据

**优先级**：P0（必须）

#### 2.2.3 数据可视化
**需求描述**：以图表形式展示监控数据趋势

**功能要求**：
- 折线图显示数据随时间变化
- X轴：时间（相对时间，单位：秒）
- Y轴：指标值（自动缩放）
- 图表交互：缩放、平移
- 多指标任务切换"指标"下拉框后，图表同步刷新为所选指标的数据
- 数据量较大时，图表数据由数据库端按分桶（最多2000桶，每桶取最小/最大值两点）降采样后返回，最多约4000个点，保留波峰波谷不失真；图表控件叠加峰值抽稀与视口裁剪进一步优化渲染（v1.2.0）

**优先级**：P0（必须）

#### 2.2.4 数据表格
**需求描述**：以表格形式展示详细数据点

**功能要求**：
- 列：时间、格式化值、原始值
- 只显示最近2000次采集的数据点（大数据量任务性能优化，v1.2.0；导出数据不受此限制，仍为全量）
- 按时间倒序排列（最新在前）
- 支持滚动查看
- 禁止编辑
- 多指标任务切换"指标"下拉框后，表格同步刷新为所选指标的数据

**优先级**：P1（重要）

#### 2.2.5 任务数据管理
**需求描述**：用户可在历史数据页面手动清理不再需要的监控任务数据，应用也可选择性地在启动时自动清理过期数据（v1.2.0）

**功能要求**：
- 历史数据页面提供"删除此任务数据"按钮，点击后二次确认，确认后删除该任务及其全部数据点
- 任务处于运行中状态时禁用删除按钮（需先停止任务），避免删除正在写入的数据引发竞态
- 支持可选的启动自动清理：通过配置项设定数据保留天数，默认0表示禁用；启用后仅清理已停止且早于保留期限的任务及其数据点，且保证在孤儿任务状态校正完成之后执行，避免误删刚异常退出的任务

**优先级**：P2（增强）

### 2.3 数据导出功能

#### 2.3.1 任务选择导出
**需求描述**：用户可以选择特定的监控任务进行数据导出

**功能要求**：
- 从下拉列表选择要导出的任务
- 显示任务详细信息（进程名、PID、监控指标——多指标显示项数与名称、开始/结束时间、采集次数与数据点总数、任务状态）
- 支持任务列表刷新
- 仅显示包含数据的任务

**优先级**：P0（必须）

#### 2.3.2 CSV文件导出
**需求描述**：将监控数据导出为CSV格式文件，多指标任务导出为宽表格式（每次采集一行，各指标各占一列）

**功能要求**：
- 支持文件保存路径选择
- 智能文件命名：`{进程名}_{指标名或"N项指标"}_{时间戳}.csv`
- CSV格式要求：
  - 使用UTF-8 BOM编码（确保Excel正确打开中文）
  - 宽表结构：前三列固定为"时间、进程名称、PID"，其后每个监控指标各占一列，列头格式为"指标名(单位)"
  - 每行对应一次采集（同一时间戳的多指标数据合并为一行），若某指标在该时间戳缺失数据则该列留空
  - 时间格式：`YYYY-MM-DD HH:MM:SS`
  - 指标数值保留4位小数
- 导出在后台线程中执行，不阻塞界面操作，支持导出中途取消（v1.2.0）；取消或失败会自动清理写了一半的文件
- 导出成功后自动打开文件所在文件夹
- 完善的错误提示和状态反馈

**优先级**：P0（必须）

#### 2.3.3 数据完整性
**需求描述**：确保导出的数据完整准确

**功能要求**：
- 导出任务的所有采集记录（含全部已选指标）
- 数值保留4位小数，缺失数据留空而非填0
- 列头本身标注指标名称与单位，无需额外的单位列
- 包含任务元信息（进程名称、PID）

**优先级**：P0（必须）

## 3. 非功能需求

### 3.1 性能要求
- 应用启动时间：< 3秒
- 数据采集延迟：< 100ms
- UI响应时间：< 200ms
- 支持长时间运行（24小时+）不崩溃

### 3.2 兼容性要求
- 操作系统：Windows 10/11 (64位)
- Python版本：3.8+
- 屏幕分辨率：1280x720 及以上

### 3.3 可靠性要求
- 进程异常终止时自动停止监控任务
- 数据库异常时给出明确错误提示
- 采集失败时不影响其他任务
- 应用崩溃前自动保存未写入数据
- 全局未捕获异常兜底：记录日志并在GUI主线程弹窗提示日志位置，工作线程异常仅记日志不打断采集；致命信号（如段错误）记录崩溃日志（v1.2.0）
- 数据落库失败自动重试，连续失败达到阈值提示用户，避免静默丢数据（v1.2.0）
- 应用意外退出（崩溃/被强制结束）后再次启动，自动将遗留的"运行中"任务状态校正为"已停止"（v1.2.0）

### 3.4 易用性要求
- 界面直观，符合Windows 11设计规范
- 关键操作有明确反馈（成功/失败/警告）
- 错误提示准确且易于理解
- 无需培训即可上手使用

### 3.5 可维护性要求
- 代码模块化，职责清晰
- 遵循PEP 8编码规范
- 关键代码有注释说明
- 提供完整的项目文档

## 6. 数据模型

### 6.1 监控任务（MonitorTask）
自v1.1.0起支持多指标监控，原单值 `metric_type` 字段已改为字符串列表 `metric_types`：
```python
{
    "task_id": "UUID字符串",
    "pid": "进程ID",
    "process_name": "进程名称",
    "metric_types": ["指标类型1", "指标类型2", "..."],  # 指标类型列表，支持多指标
    "interval": "采集间隔(秒)",
    "start_time": "开始时间",
    "end_time": "结束时间(可空)",
    "status": "running | stopped"
}
```

### 6.2 数据点（DataPoint）
数据点新增 `metric_type` 字段，用于标记该数据点归属于任务的哪一项指标（空串兼容旧数据）：
```python
{
    "task_id": "所属任务ID",
    "timestamp": "时间戳",
    "value": "指标值(浮点数)",
    "metric_type": "指标类型（空串表示旧版无该字段的历史数据）"
}
```

### 6.3 数据库结构与迁移

**表结构**：
- `tasks` 表：`metric_type` 列存储 JSON 数组文本（如 `["memory_rss","cpu_percent"]`），对应 `metric_types` 字段
- `data_points` 表：新增 `metric_type` 列（允许为 NULL，兼容升级前采集的历史数据）
- 新增复合索引 `idx_data_points_task_metric (task_id, metric_type)`，用于按任务+指标查询
- 每个数据库连接开启时统一设置 `WAL` 日志模式、`busy_timeout=5000ms`、`synchronous=NORMAL`，提升并发读写下的稳定性（v1.2.0）

**自动迁移机制**：
- 数据库使用 `PRAGMA user_version` 标记 Schema 版本（当前为 v1）
- 应用启动打开数据库时，若检测到 `user_version` 低于当前版本，自动执行迁移：先给 `data_points` 补充 `metric_type` 列并按任务原有指标回填历史数据点，再将 `tasks.metric_type` 由裸字符串转换为 JSON 数组
- 迁移前先对旧库做一次 WAL checkpoint（把未落盘的写日志合并进主文件），再将数据库文件自动备份为同目录下的 `monitor.db.bak_v0`，原始数据永不删除（v1.2.0：备份前补充 checkpoint，避免备份遗漏尚在日志中的数据）
- 迁移在单个数据库事务中执行，失败会自动回滚；若迁移失败会先从备份文件还原一次并重试，两次仍失败则将损坏的数据库文件改名保留（形如 `monitor.db.broken_时间戳`）、新建空库
- 迁移结果分三种明确状态，分别对应不同的主界面提示（v1.2.0）：
  - **迁移失败已还原**（`migration_failed`）：两次迁移均失败，已还原旧数据，提示用户本次会话新数据无法保存，需重启重试
  - **数据已重置**（`data_reset`）：还原备份也失败，损坏库已改名保留，应用以新建空库运行，提示用户历史数据已移至 `monitor.db.broken_*`
  - **迁移已中止**（`backup_aborted`）：迁移前备份本身失败（如磁盘空间不足），直接中止迁移、旧库原样保留未做任何改动，提示用户释放磁盘空间后重启
- 整个迁移流程每次应用运行只尝试一次，避免同一进程内因反复实例化数据库对象而重复触发备份

## 8. 异常处理

### 8.1 进程不存在
- **场景**：用户输入的PID不存在或无法访问
- **处理**：显示错误提示，禁止创建任务

### 8.2 进程已终止
- **场景**：监控过程中进程意外终止
- **处理**：自动停止监控任务，保存已采集数据，提示用户

### 8.3 数据库异常
- **场景**：数据库文件损坏或权限不足
- **处理**：显示错误信息，尝试重新初始化数据库
- **场景**：单次数据保存失败（磁盘瞬时繁忙等）
- **处理**：自动保留待落库数据并在下一采集周期重试，不立即丢弃；连续失败达到阈值时提示用户一次（v1.2.0）
- **场景**：旧版本数据库升级迁移失败（详见6.3节）
- **处理**：自动从迁移前备份还原并重试一次；三种最终结果对应三种明确的主界面提示——迁移失败已还原（提示重启重试）、数据已重置（提示历史数据已移至`.broken_*`文件）、迁移已中止（提示释放磁盘空间后重启）（v1.2.0）

### 8.4 任务数量超限
- **场景**：用户尝试创建第6个监控任务
- **处理**：显示警告提示，禁止创建
